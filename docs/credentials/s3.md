# Amazon S3 (and S3-compatible) credentials

The S3 provider works with AWS S3, Cloudflare R2, Wasabi, DigitalOcean
Spaces, MinIO, Backblaze B2 (via S3 API), Scaleway Object Storage, and
anything else that speaks the S3 wire protocol.

What you need before running the bootstrap:

- An existing bucket
- An access key ID + secret access key with **read/write** on that bucket
- The region the bucket lives in
- (For non-AWS) the endpoint URL

## AWS S3 — the specific walkthrough

### 1. Create a bucket

1. Sign in to <https://console.aws.amazon.com/s3/>.
2. Create bucket → name it `yourname-vps-backups` (bucket names are global).
3. Pick a region — write it down, you'll paste it during bootstrap.
4. **Block all public access** — leave checked. Backups are private.
5. Enable **versioning** if you want protection against ransomware / accidental deletes.
6. Create bucket.

### 2. Create a least-privilege IAM user

Don't use your root access key. Make a dedicated one:

1. <https://console.aws.amazon.com/iam/> → **Users** → **Create user**.
2. Name: `vcb-backup-USER` (pick something you'll recognize).
3. **Attach policies directly** → **Create policy**. Paste this JSON,
   replacing `yourname-vps-backups` with your actual bucket:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListBucket",
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::yourname-vps-backups"
    },
    {
      "Sid": "ObjectOps",
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject",
        "s3:AbortMultipartUpload",
        "s3:ListMultipartUploadParts"
      ],
      "Resource": "arn:aws:s3:::yourname-vps-backups/*"
    }
  ]
}
```

4. Save the policy. Back on the user's **Permissions** tab, attach it.
5. **Security credentials** tab → **Create access key** → pick "CLI".
6. Copy the **Access key ID** and **Secret access key**. You will NOT
   be able to see the secret again — write it somewhere safe.

### 3. Run the bootstrap

During `phase_4_provider_prompt`, pick `s3`. Answer:

```
rclone remote name         [s3]
S3-compatible provider     AWS
S3 region                  us-east-1     (whatever you picked above)
S3 endpoint URL            (leave blank for AWS)
S3 bucket name             yourname-vps-backups
S3 access key ID           AKIA...
S3 secret access key       ...            (hidden while typing)
```

## Non-AWS providers

Same flow, different answers at the prompts:

| Provider     | Provider key | Region  | Endpoint                                |
|--------------|--------------|---------|------------------------------------------|
| Cloudflare R2| Cloudflare   | auto    | https://<account>.r2.cloudflarestorage.com |
| Wasabi       | Wasabi       | (per region) | https://s3.wasabisys.com              |
| DO Spaces    | DigitalOcean | (per region) | https://<region>.digitaloceanspaces.com |
| MinIO        | Minio        | us-east-1 | http://minio.example.com:9000            |
| Backblaze B2 | Other        | us-west-002 | https://s3.us-west-002.backblazeb2.com |

Credentials are always the same pair (access key ID + secret access
key), minted in each provider's console.

## Verify

After bootstrap finishes:

```
rclone lsd s3:yourname-vps-backups
```

should show at least the `backups/<hostname>_<ip>/` prefix the bootstrap
created (or be empty on a brand new bucket).
